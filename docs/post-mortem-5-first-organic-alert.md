## Postmortem: First Organic Budget Alert

Project: Cost Sentinel (AWS Budget → SNS → Lambda → S3 Dashboard)
Date: 2026-06-05
Status: Validated – System caught real spend without intervention

---

## Summary

After completing active development and security hardening, Cost Sentinel transitioned to passive monitoring of a live AWS account. On 2026-06-04, a Route 53 domain registration generated unexpected spend that pushed the account's monthly forecast above the $10 threshold. Cost Sentinel detected it and delivered a budget alert notification the following day.

This is distinct from the synthetic validation during initial development, where Bedrock spend from the infra-agent project was used to confirm the pipeline worked end-to-end. That was an engineered test. This was organic: the system caught real spend on its own while running unattended in the dev environment.

The event validates that Cost Sentinel works as intended in steady-state operation and surfaces a clear next step: promote the system to a production environment.

---

## What Happened

| Timestamp | Event |
|-----------|-------|
| 2026-06-04 | Route 53 domain registration incurred a one-time charge |
| 2026-06-05 | AWS Budgets projected monthly spend would exceed the $10 threshold |
| 2026-06-05 | Budget alert fired → SNS → notification delivered |
| 2026-06-05 | Investigation confirmed Route 53 as the source; no other anomalies found |

---

## Context: Synthetic Validation vs. Organic Detection

During initial development, the pipeline was validated intentionally using Bedrock API spend generated by a separate project (infra-agent). That confirmed the mechanics worked — budget threshold, SNS topic, Lambda ingestor, notification delivery — but it was a controlled test, not a real catch.

Since completing that phase, Cost Sentinel has been running unattended. This alert is the first time the system independently detected spend that the account owner did not proactively track. That is the intended behavior.

---

## Root Cause of the Spend

Route 53 domain registration (one-time fee). Not a misconfiguration or runaway resource — a deliberate but routine action that happened to cross the monitoring threshold.

No remediation required.

---

## What Went Well

- Alert fired without any manual intervention or prompting
- Notification delivered successfully through the full pipeline
- Root cause was identifiable in under a minute from the alert
- The $10 threshold proved appropriately sensitive for a personal AWS account — low enough to catch incidental spend, high enough to avoid noise from baseline infrastructure costs

---

## Observations

The dev environment has been the only deployed environment since the project began. It has served as both the development target and the live monitoring system for the account. That conflation made sense during active development but is worth revisiting now that the system is proven.

A production environment would:
- Separate monitoring infrastructure from the environment being actively changed
- Reduce the risk of a Terraform apply or pipeline run disrupting alert delivery
- Allow dev to be iterated on (or torn down) without losing monitoring coverage
